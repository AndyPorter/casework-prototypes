Case list

Service Design Assumptions/Parameters

  • We are getting mortgage applications from a few, known, lenders
  • Number of applications will be small.
  • Internal processing team will be small, co-located, and co-located with the service development team - operate, iterate and support the application/service as one team.
  • we will be iterating/bug fixing/enhancing on a daily basis
  • If the new system/process cannot cope with an application we can run it through the existing systems/processes
  • first release could look / operate nothing like later releases (metamorphosis).
  • an aim (not the only) of first release is to learn what we can take away - what isn't technical work/legal judgement - to be able to remove that and leave technical skilled work for staff.

Design Principles / User needs/desires

  • minimal interaction with existing applications and processes - only where we absolutely have to
  • Everything needed to operate the application is brought to the user
  • Users are legal technicians
  • tools focused on the goal - processing mortgage appication (not a swiss army knife system)
  • simplify language. clarity and accuracy of terminololgy
  • less is best - strip away everything not needed. start with too little and add.
  • more interesting work / less admin
  • confidence / trust in the systems. IT and in what we are asking them to do (process/practice)
  • supported and listened to
